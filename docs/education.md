## 1. 背景：问题是什么

我们要训练一个神经网络，让它看着摄像头画面和机械臂当前关节角度，输出下一步要执行的**动作**（6个关节角度 + 1个夹爪开度），从而完成「抓瓶子」这个任务。

```
摄像头画面 ──┐
              ├──→ 神经网络 ──→ 7维动作（关节目标位置）
机械臂状态 ──┘
```

**模型架构**：ACT（Action Chunking Transformer），一个 Transformer 编解码器，一次预测 10 步动作序列。

**训练数据**：50 条人工遥控的抓取轨迹，30fps，每条约 200 帧，共 10926 帧。

---

## 2. 第一步：模型为什么会「躺平」

### 现象

第一次训练完成后（3万步），我们把模型部署到真机上，发现机械臂**完全不动**——无论摄像头看到什么，模型输出的目标位置都是一个固定值：

```
Target (恒定): [0.414, 1.629, -1.173, -0.024, 0.396, -0.191, 0.035]
```

机械臂从起点直接飞到这个固定位置然后卡住。

### 根因分析

这叫做**策略坍缩（Policy Collapse）**。根本原因有二：

**原因 1：绝对动作 + L2 损失 = 回归均值**

模型被训练来预测 **绝对关节目标位置**（`目标 = 模型输出`），损失函数是 L2（均方误差）。

但问题在于：同一个输入（比如「瓶子在面前」的画面），在不同的轨迹阶段，正确答案完全不同——刚开始应该伸手靠近，快结束时应该合拢夹爪。

当模型面对这种**一对多**映射（相同输入 → 不同输出），L2 损失的最优解就是输出所有可能答案的**平均值**。平均值只有一个，于是模型对所有输入输出同一个固定值——「躺平」了。

```
输入 A ─→ 正确答案可以是 动作X 或 动作Y
输入 B ─→ 正确答案可以是 动作P 或 动作Q

模型学到的: 对所有输入都输出「平均动作」
```

**原因 2：确定性 Transformer 没有随机性**

VAE（变分自编码器）可以通过采样引入多样性，但我们关闭了 VAE（`use_vae=false`）以加速训练。确定性模型没有随机性来源，一旦坍缩就无法自救。

### 可视化

![训练分析](fig_training_analysis.png)

上图右侧对比了坍缩模型和修复后模型的关节运动范围。红色柱（旧模型）几乎为零——所有关节输出固定值。蓝色柱（修复后）与绿色柱（真值）量级接近，说明模型在输出有意义的变化。

## 3. 第二步：Delta Action — 从「走到终点」变成「走一小步」

### 思路

不让模型直接预测「目标位置在哪」，而是预测「**从当前位置应该往哪挪一小步**」。

```
旧（Absolute）: 模型输出 = 目标关节角度 target
               机械臂: 直接走到 target

新（Delta）:    模型输出 = delta（位移量）
               机械臂: target = 当前位置 + delta
```

### 为什么能解决坍缩

如果模型「躺平」了，输出全零 delta，机械臂只是**原地不动**——不会飞到奇怪的位置。这是安全的失败模式。

而且 Delta 的均值天然接近零（轨迹中有正有负），所以即使模型回归均值，也是回归到接近零的 delta，机械臂几乎不动，不会造成危险。

### 数据改造

训练数据原本存储的是绝对动作 `action = state[t+5]`，需要改为 delta：
```
delta_action = state[t+5] - state[t]
```

这样模型学习的就是「未来 5 帧的关节变化量」。

## 4. 第三步：Phase 信号 — 告诉模型「你走到哪了」

### 问题

即使改成 Delta Action，模型仍然面临**时序歧义**：

```
机械臂看到: 关节=[0.3, 0.5, -0.8, ...], 画面=瓶子在夹爪附近
可能阶段 A: 正在靠近瓶子（需要继续前进）
可能阶段 B: 已经抓到准备后退（需要合拢后退）
```

同一个状态，在轨迹的不同阶段，正确答案不同。

### 方案

在输入中增加一个 **Phase（进度）信号**：一个 0→1 的浮点数，表示当前在整条轨迹中的进度。

```
Phase = 当前帧序号 / 轨迹总帧数

Phase ≈ 0.0  →  刚开始，模型应该输出「靠近瓶子」的 delta
Phase ≈ 0.5  →  中途，正在调整姿态
Phase ≈ 1.0  →  快结束，应该抓取并抬起
```

现在输入从 7 维变成 8 维：`[j1, j2, j3, j4, j5, j6, gripper, phase]`

Phase 打破了时序歧义——同一个画面和关节状态，配合不同的 phase 值，模型就能输出不同的 delta。

## 5. 第四步：Replan Every Step — 走一步看一步

### 问题

ACT 设计是一次性输出 10 步动作（chunk_size=10），然后机械臂连续执行这 10 步，中间不再观察。这是**开环控制**。

如果 10 步中环境变化（瓶子被碰动、机械臂偏了），模型看不到。

### 方案

设置 `n_action_steps=1`，即每次只执行 1 步，然后**重新观察、重新推理**。

```
旧: 观察 → 规划10步 → 闷头走10步 → 观察 → 规划10步 → ...
新: 观察 → 规划1步 → 走1步 → 观察 → 规划1步 → ...
```

这本质上是把开环系统变成了**闭环系统**。chunk_size=10 仍然保留，让 Transformer 有 10 步的时间上下文来理解轨迹走向，但我们只取第一个动作执行。

## 6. 第五步：数据集重建

用 `scripts/rebuild_delta_phase_dataset.py` 将原始的绝对动作数据集转换为 Delta+Phase 格式：

| 字段 | 旧格式 | 新格式 |
|------|--------|--------|
| `observation.state` | `[j1..j6, gripper]` (7D) | `[j1..j6, gripper, phase]` (8D) |
| `action` | `[j1..j6, gripper]` 绝对目标 | `[dj1..dj6, dgripper]` Delta位移 |

数据集统计：
- 50 条轨迹，共 10926 帧
- 轨迹长度：最短 185 帧，最长 264 帧，平均 219 帧
- 帧率：30fps
- 每条轨迹约 6~9 秒

## 7. 第六步：单集过拟合 — 管线最低门槛测试

### 为什么必须做

在投入数小时全量训练之前，先用**一条数据**验证整个管线是否正确。这是整个项目成本最低、收益最高的检查。

```
单集过拟合的核心逻辑：

  如果模型连「背答案」都做不到（1 条数据 loss 降不下去）
  → 那训练 50 条也一定学不会
  → 问题一定在基础设施上（数据格式、维度匹配、loss 计算）

  如果一条数据能完美拟合（MSE → 0）
  → 至少说明整个管线没有低级 bug
  → 可以放心开始全量训练
```

### 实际操作

用 `scripts/rebuild_delta_phase_dataset.py` 从完整数据集中拆出 episode 0：

```bash
python3 scripts/rebuild_delta_phase_dataset.py \
    --input-root data/lerobot_dataset_v2 \
    --output-root data/lerobot_dataset_v2_delta_ep0 \
    --delta-horizon-frames 1 --no-phase \
    --episode 0 --overwrite
```

训练脚本 `training/train_act_delta_v2_overfit_ep0.sh` 关键参数：

| 参数 | 单集值 | 全量值 | 原因 |
|------|--------|--------|------|
| batch_size | 2 | 4 | 单集数据少，小 batch 更新更频繁 |
| dropout | 0.0 | 0.1 | 只有一条数据无需防过拟合 |
| image_transforms | false | true | 关闭数据增强，免除干扰 |
| steps | 5000 | 30000 | 单条数据收敛快，几千步就够 |
| 耗时 | ~8 分钟 | ~80 分钟 | — |

### 判决标准

跑完用 `scripts/check_policy_collapse.py` 检测：

| 指标 | 通过标准 | 本次实际值 |
|------|----------|-----------|
| collapsed | **False** | False |
| MSE | < 0.01 | 0.000048 |
| pred arm range | 与 GT 量级一致，不能全为零 | [0.006~0.067]，量级正常 |
| chunk arm range | > 0.01 | 0.050 |

**如果 collapsed=True 或 MSE > 0.1，禁止进入全量训练。** 先用 `scripts/deep_diagnosis.py` 排查，直到通过为止。

### 常见失败原因

| 现象 | 可能原因 |
|------|----------|
| MSE 不下降（> 1.0） | 数据维度不匹配（state 8D vs action 7D） |
| collapsed=True | 绝对动作模式，模型回归到固定值 |
| pred range 全是零 | action 标签恒为零（delta 计算错误） |
| 单集通过但很慢（> 5000 步才收敛） | learning rate 太低或模型容量过大 |

### 教训

> 不要在没跑过单集过拟合之前开始全量训练。花 8 分钟做这个检查，可以避免浪费一小时的 GPU 时间和更多的心力。

## 9. 第八步：坍缩检测（上线前必查）

用 `scripts/check_policy_collapse.py` 在数据集上跑推理，自动判断是否坍缩：

```
判断标准:
  pred_step.max() < 1e-4  且  chunk_ranges.max() < 1e-4
  → collapsed = True（坍缩，不可部署）

  pred_step.max() > 1e-4  或  chunk_ranges.max() > 1e-4
  → collapsed = False（正常，可以部署）
```

### 检测结果

| Checkpoint | MSE | pred_step max | chunk_range max | Collapsed | 可部署 |
|-----------|------|---------------|-----------------|-----------|--------|
| 20,000 步 | 0.001262 | 0.121 | 0.211 | **False** | ✓ |
| 30,000 步 | 0.001378 | 0.174 | 0.235 | **False** | ✓ |

两个 checkpoint 均通过检测。

## 10. 第九步：真机部署与失败案例

### 部署命令

```bash
python3 inference/deploy.py \
    --checkpt outputs/train/piper_bottle_grasp_delta_phase/checkpoints/030000/pretrained_model \
    --action-mode delta \
    --debug-actions --debug-every 10 \
    --replan-every-step \
    --hz 30 \
    --max-steps 220
```

### 失败案例 1：Phase 不匹配导致戳海绵

**现象**：机械臂直接向下戳到了工作台海绵里。

**原因**：`--max-steps` 使用了默认值 500，但训练数据平均每条约 219 帧。Phase 计算公式为 `step / max_steps`，当 max_steps=500 时，phase 每步只增加 0.002，而训练时每步增加约 0.005。

这导致模型一直处于「轨迹早期」状态——它以为还需要大步靠近，于是持续输出较大的向下 delta，最终戳进海绵。

**教训**：`--max-steps` 必须匹配训练数据集的平均轨迹长度。对于 30fps、平均 219 帧的数据集，max_steps 设为 220。

### 失败案例 2：绝对动作模型的完全坍缩

**现象**：模型输出固定值，机械臂飞到固定位置后不动。

**坍缩目标值**：`[0.414, 1.629, -1.173, -0.024, 0.396, -0.191, 0.035]`

**检测脚本确认**：`collapsed: True`，pred_step.max ≈ 0.0，chunk_ranges.max ≈ 0.0。

**教训**：绝对动作 + 确定性 Transformer = 必然坍缩。必须使用 Delta 动作或 VAE 随机性来防止。

## 11. 总结：关键教训

| # | 教训 | 严重程度 |
|---|------|----------|
| 1 | **绝对动作 + L2 损失 → 策略坍缩**，确定性 Transformer 输出均值 | 致命 |
| 2 | **Delta 动作**是性价比最高的防坍缩方案：即使摆烂也安全 | 关键 |
| 3 | **Phase 信号**打破时序歧义，同样输入可以产出不同输出 | 重要 |
| 4 | **Replan Every Step** 将开环变为闭环，逐帧纠正偏差 | 重要 |
| 5 | **部署前必须跑 collapse check**，否则可能白跑一趟真机 | 必须 |
| 6 | **max_steps 要匹配训练数据**，否则 phase 编码对不上 | 容易忽略 |
| 7 | **单集过拟合是全量训练的前置条件**，不过关就开工等于闭眼开车 | 必须 |
| 8 | **单条过拟合测试**只能验证管线贯通，不能代表泛化能力 | 易误解 |

### 完整管线

```
数据采集 ──→ 重建Delta+Phase数据集 ──→ 训练(30K步) ──→ 坍缩检测 ──→ 真机部署
  ↑                                                                    │
  └────────────────────  失败反馈，迭代改进  ←──────────────────────────┘
```
